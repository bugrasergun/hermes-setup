# Brain Search Infrastructure Details

## Database: `brain_documents` Table (Layer 2 + 3)

```sql
CREATE TABLE brain_documents (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    file_path TEXT NOT NULL,
    file_hash TEXT NOT NULL,          -- SHA-256 of file content
    chunk_index INTEGER NOT NULL,     -- 0-based chunk order within file
    content TEXT NOT NULL,            -- chunk text (markdown, extracted PDF, transcript)
    embedding vector(768),           -- nomic-embed-text embedding
    agent_profile TEXT NOT NULL DEFAULT 'shared',   -- 'shared', 'ayda', 'alex', 'sage', 'maya', etc.
    source_type TEXT NOT NULL DEFAULT 'markdown',   -- 'markdown', 'pdf', 'txt', 'transcript'
    page_num INTEGER DEFAULT NULL,                   -- PDF page number for citation
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE(file_path, chunk_index)
);
```

**Indexes:**
- `ix_brain_documents_embedding_hnsw` — HNSW on `embedding` with `vector_cosine_ops` (m=16, ef_construction=64)
- `ix_brain_documents_file_path` — btree for file lookups
- `ix_brain_documents_file_hash` — btree for change detection
- `ix_brain_documents_agent_profile` — btree for per-agent filtering

**New columns (Layer 3, 2026-07-08):**
- `agent_profile` — derived from path (`references/sources/<name>/` → `<name>`, else `shared`)
- `source_type` — file extension mapped: `.md`=markdown, `.pdf`=pdf, `.txt`=txt, `.srt`/`.vtt`=transcript
- `page_num` — for PDF chunks (fingerprint-matched against per-page text)

## Supported File Types

| Type | Extensions | Extraction | Chunking |
|------|-----------|------------|----------|
| Markdown | `.md` | Direct read | Header-based |
| PDF | `.pdf` | PyMuPDF4LLM → markdown | Header-based + page tracking |
| Text | `.txt` | Direct read | Header-based |
| Transcript | `.srt`, `.vtt` | Timestamp stripping | Header-based |

## Chunking Strategy

**Method:** Markdown header-based (`#`, `##`, `###` split points)

1. Split content at every header line (`^(#{1,6}\\s+.+)$`)
2. Each chunk retains its header for context
3. Content before the first header (if meaningful) becomes its own chunk
4. Sections > 1500 chars are split further with 150-char overlap
5. Break priority within long sections: paragraph (`\\n\\n`) → sentence (`. `) → hard cut
6. Chunks < 80 chars are skipped (too small for meaningful embedding)

**PDF chunking:** PyMuPDF4LLM converts PDF to markdown, preserving tables and formatting.
Then header-based chunking. Page numbers are assigned via fingerprint matching — the first
50 characters of each chunk are matched against per-page text (extracted by PyMuPDF).

**Transcript chunking:** SRT/VTT files are stripped of timestamps and sequence numbers.
Resulting plain text goes through the same header-based chunking.

## Profile Derivation

Agent profile is derived from file path:

```
references/sources/shared/file.pdf     → agent_profile = 'shared'
references/sources/maya/fitness.pdf    → agent_profile = 'maya'
projects/taco-bros/strategy.md         → agent_profile = 'shared'  (all brain vault)
```

## Search Query (with profile filter)

```sql
-- Default: shared only
SELECT file_path, chunk_index, content, agent_profile, source_type, page_num,
       1 - (embedding <=> :query_embedding) as similarity
FROM brain_documents
WHERE embedding IS NOT NULL AND agent_profile = 'shared'
ORDER BY embedding <=> :query_embedding
LIMIT :top_k;

-- Agent-specific: agent's own + shared
SELECT ... WHERE embedding IS NOT NULL AND agent_profile IN (:profile, 'shared') ...

-- All: everything
SELECT ... WHERE embedding IS NOT NULL ...
```

## Performance

- Full rebuild (126 files, ~733 chunks): ~53 seconds (Ollama embedding is the bottleneck)
- Update (changed files only): ~0.1s per file check, ~2s per PDF chunk (extraction)
- Search: ~0.3s (HNSW index makes it fast)
- PDF extraction overhead: ~3-5s/page first time (PyMuPDF4LLM conversion)

## SQL Parameter Pinning for `--profile`

When modifying the `search_brain()` SQL for agent-specific profile filtering,
never use a `%s` placeholder for the profile name inside `IN (...)`:

```python
# ✅ CORRECT — f-string with repr for safe SQL injection
sql = f"""SELECT ... WHERE agent_profile IN ({profile!r}, 'shared')
ORDER BY embedding <=> %s::vector LIMIT %s;"""
params = (emb_str, emb_str, top_k)

# ❌ WRONG — psycopg counts %s strictly; 5 params for 4 placeholders breaks
sql = """... WHERE agent_profile IN (%s, 'shared') ... ORDER BY embedding <=> %s::vector LIMIT %s;"""
params = (emb_str, profile, emb_str, emb_str, top_k)
```

The profile value comes from a trusted source (CLI arg, not user data), so
string interpolation with `{profile!r}` is safe here.
